## Project State

- The site is a single-page Next.js App Router experience with site chrome, hero, about, services, pricing, testimonials, contact, and back-to-top behavior.

## Current Repository State

- Next App Router entrypoints are `app/layout.tsx` and `app/page.tsx`.
- `app/page.tsx` composes the single `/` page in section order and exports static route guardrails so build-time prerendering fails loudly if request-time rendering is introduced. [Reason why added: production should keep the Next runtime while returning the root page from prebuilt static output.]
- Browser-dependent behavior is isolated in client components for header/nav state, Base UI mobile menu dialog behavior, hero CTA annotation, testimonials carousel/dialog behavior, shared CTA buttons, and the back-to-top widget.
- UI source is organized by ownership: page sections live under `src/sections`, site chrome lives under `src/layout`, reusable primitives live under `src/shared/ui`, and app-level widgets live under `src/widgets`.
- Section folders contain React section components; shared and feature-specific non-component utilities live in folders under `src/utils/<utility-area>`. [Reason why added: GSAP timelines, easing functions, and formatting helpers are app utility logic, not React components, and should not clutter section component folders or the root utility folder.]
- Most UI files pair `component-name.tsx` with `component-name.module.css`.
- Ordinary frontend source filenames use lowercase kebab-case; React symbols remain `PascalCase`.
- The global CSS entrypoint is `app/globals.css`; shared global styles live in `src/styles`, route-local styles live beside their route files, and local Inter font files are loaded through `next/font/local`.
- The shared color system is centralized in `src/styles/vars.css` as OKLCH CSS custom properties, with repeated semantic text tokens aliasing canonical palette tokens where possible. [Reason why added: KAN-146 migrated the previous HSLA/RGBA color representations to OKLCH without intentionally changing the visual palette.]
- Styling follows a mobile-first approach: base styles target the smallest supported layout, with larger layouts layered through responsive overrides. The canonical viewport tiers are mobile at `width <= 576px`, tablet from `577px` through `1160px`, desktop from `1161px`, and extra-wide from `1921px`; matching container tokens in `src/styles/vars.css` are `--container-mobile: 432px`, `--container-tablet: 720px`, `--container-desktop: 1300px`, `--container-wide: 1600px`, and `--container-extra-wide: 1920px`. The back-to-top widget additionally uses a local `1761px` threshold for placement. Media queries use modern range syntax (e.g., `width >= 577px`). [Reason why added: future responsive changes need one traceable breakpoint contract instead of rediscovering media-query thresholds from scattered CSS Modules, image `sizes`, and GSAP matchMedia strings.]
- Normal section anchor offsets are aligned with normal section vertical margins by breakpoint, while controlled site-nav scrolling reserves the fixed dockbar height plus matching top/bottom visual gap when navigating upward or loading section hashes. The Hero nav item points to `/` and scrolls to the true page top; legacy `/#hero` links are normalized back to `/` after scrolling to top. Hero nav active state is derived from being above the About section. [Reason why added: the site canonical URL is the root page, real section hashes remain useful deep links, and the Hero item represents page home/top rather than a separate section anchor.]
- The hero keeps viewport-filling minimum height on mobile/tablet and uses viewport-aware minimum height floors from `min-width: 1161px` upward, with the desktop main content row flexing to absorb extra vertical space while CTA and logo rows remain fixed-size. Desktop and extra-wide hero formulas align the first screen to the viewport so the next section does not peek through; the visible hero edge spacing targets `40px` on ordinary desktop and `56px` at `1921px+`, and the extra-wide hero footprint is capped at `1600px` including the header and hero top margins. At `1921px` and wider, root spacing and font tokens follow a consistent extra-large scale while the global section max-width layer remains capped by `--container-wide` through `--section-max-extra-wide`; hero-only slider slots, note size, an extra-wide flex-growing portrait lane with direct container-query-based image scaling and a larger profile-block gap, larger heading-to-content gaps, full-width About row with a badge-aware photo lane, large button sizing, Contact image/row scale, Contact CTA equal-width action stack, equal-height content-sized Services card tracks, content-sized Pricing cards, fixed testimonial card heights, extra-wide pricing-card width, and tutor-pricing action row alignment are adjusted to match the scaled text, while Services/Pricing `h3` subtitles are not capped to copy-wrapper widths. [Reason why added: KAN-131 keeps the hero organized across tall screens by filling available viewport height through structural flex space instead of arbitrary loose gaps, while preserving systematic extra-large typography, gaps, containers, buttons, images, and fixed-format UI elements.]
- `src/styles/animations.css` owns shared global keyframes and exposes animation-name custom properties for CSS Modules that need to reference those keyframes. [Reason why added: Next CSS Modules scope direct keyframe identifiers in module CSS, so shared animations must cross that boundary through explicit global tokens.]
- The hero role word, note card, and animated stat are coordinated by a scoped `@gsap/react` timeline in a client wrapper using transform-based movement, mask-softened text edges, shared phase timing, and a reduced-motion fallback. [Reason why added: these hero highlights need to start and finish together instead of relying on independent CSS keyframes that can drift or animate layout properties.]
- The hero logo marquee is a client component that renders one accessible set of logos plus hidden visual repeats, and uses `@gsap/react`, `gsap.matchMedia()`, and a GSAP timeline to wrap individual logo items horizontally with width-aware `xPercent` transforms while respecting `prefers-reduced-motion` and pausing for non-touch pointer/focus interaction. [Reason why added: duplicated CSS tracks could show a shift or blink at the loop reset seam on Android Chrome and iPadOS Safari, while a single visual set could make returning items appear inside ultra-wide viewports instead of entering from the right edge.]
- The hero logo marquee and testimonials carousel share GSAP responsive pixel-per-second targets from `src/utils/gsap/responsive-marquee-speed.ts`: 20px/s on mobile, 28px/s on tablet, and 44px/s on desktop. [Reason why added: a fixed pixel speed feels faster on small screens because it crosses a larger share of the viewport per second, while desktop needs a higher raw speed to avoid feeling stalled.]
- Infinite GSAP marquee timelines are viewport-gated through the shared marquee utilities, and the testimonials carousel keeps both visual rows interactive while reapplying modal, non-touch pointer, and keyboard-focus pause state when responsive timelines rebuild. Dialog close handling preserves the pause while the cursor remains over the testimonials row. [Reason why added: continuously repeating transform timelines should not keep compositor work active while offscreen, visible duplicate testimonial cards must not expose dead `Read more` controls, and user-paused carousel content should stay paused through media-query changes.]
- Images, fonts, and icons live under `src/assets`; public web assets live under `public`.
- Raster UI images render through `next/image` with `quality={100}` allowed by `next.config.ts`; the hero/about portraits use mobile assets as the default source and tablet assets from `min-width: 577px` through `getImageProps`/`picture` art direction, and the About portrait and Contact artwork have desktop/extra-wide CSS sizing plus matching image `sizes` for their scaled presentations. Services, Pricing, and Testimonials intro illustrations are capped to their `368px` source width at `1921px+` with matching `sizes` hints, so extra-wide layouts do not upscale those low-resolution PNGs; true retina sharpness still needs higher-resolution source assets. Height-constrained decorative images explicitly keep `width: auto` so Next's intrinsic image attributes do not distort aspect ratios. SVG logo/icon assets are rendered deliberately through `next/image` with per-image SVG optimization disabled, and the shared brand logo does not request image preload. [Reason why added: default Next image quality, lost mobile-first portrait source selection, height-only CSS, and extra-wide upscaling of 368px section illustrations reduced visual fidelity after the KAN-110 migration and later responsive scale changes.]
- `public/robots.txt` allows crawlers to access only the exact root route plus the static Next/image and metadata assets needed to render and preview that route; all other routes are disallowed by default. [Reason why added: the site is a single-page public experience and should not invite crawling of non-root route paths.]
- Vercel Analytics is rendered from `app/layout.tsx` for production builds only, keeping the local dev CSP free of the Analytics debug-script warning. [Reason why added: the dev-only Vercel Analytics debug script is externally hosted and blocked by the project's local CSP.]
- There is no configured test script in `package.json`.
- Project-support files now live under `.agents/project-files`.
- Repository line endings are defined by `.gitattributes` with LF for text files and binary handling for images/fonts.
